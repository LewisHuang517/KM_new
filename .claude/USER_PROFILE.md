## 🗣️ 溝通風格深度分析

### 語言與表達特徵
- **主要語言**：繁體中文（台灣）
- **句子長度**：極短，平均 5-15 字
- **標點符號**：少用，只在必要時使用
- **錯字容忍度**：高（效率優先，不影響理解即可）
  - 範例：「席官」（細節）、「部一樣」（不一樣）
- **語氣**：平實、不客套、不過度禮貌

### Prompt 模式識別

#### 類型 A：確認式（問題已解決）
```
✓ "OK，問題解決"
✓ "OK了目前沒問題"
✓ "我確認完程式碼可以運行了，繼續教我吧"
```
**AI 應對**：立即進入下一個任務，不要追問「還有什麼問題嗎」

#### 類型 B：需求式（明確指令）
```
✓ "幫我更改歷史影片檢視的排版，讓這個頁面更加直觀好操作"
✓ "幫我把那些臨時測試用的資訊都刪除"
✓ "教我怎麼git push到 github"
```
**AI 應對**：直接執行，少問多做

#### 類型 C：問題式（發現異常）
```
✓ "我發現每個影片的縮圖都是一模一樣的"
✓ "為什麼上面已經連接NAS了為什麼下面還會再連接一次"
```
**AI 應對**：先分析原因，再提供解決方案

#### 類型 D：深度分析式（ultrathink 標記）
```
✓ "我發現每個影片的縮圖都是一模一樣的 ultrathink"
✓ "盡量幫我把細節描述得更清楚一些好了 ultrathink"
```
**AI 應對**：進行全面、深入的分析，提供完整的技術細節和多種方案

#### 類型 E：安全確認式（擔心破壞性操作）
```
✓ "那些東西刪掉沒關係嗎"
```
**AI 應對**：清楚解釋影響，先說明安全性再執行

#### 類型 F：自主決定式（拒絕協助）
```
✓ "沒關係我自己來好了"
```
**AI 應對**：提供簡短提示後退出，不要堅持幫忙

### 反饋習慣
1. **成功**：「OK」、「OK了」、「問題解決」
2. **部分成功**：會詳細描述哪些成功、哪些失敗
   - 範例：「有一些有縮圖了，並且不一樣，不過我透過搜尋搜尋出來的影片有些縮圖跑不出來」
3. **失敗**：直接貼上錯誤日誌或終端輸出
4. **打斷**：看到 AI 說太多會打斷（[Request interrupted by user]）

### 溝通偏好總結
| 特徵 | 偏好 | 不喜歡 |
|------|------|--------|
| 回應長度 | 簡短、分點列出 | 長篇大論 |
| 選擇題 | 2-3個選項，快速決定 | 超過4個選項 |
| 解釋深度 | 關鍵概念+範例 | 過度學術化 |
| 執行方式 | 直接幫我做 | 只給建議不執行 |
| 確認頻率 | 危險操作才問 | 每個步驟都確認 |

### 程式碼品質要求

#### 整潔度標準
1. **絕對不允許**：
   - `console.log()` 除錯訊息（完成後必須刪除）
   - 註解掉的舊代碼
   - 臨時測試用的變數或函數
   - 無用的 import

2. **必須做到**：
   - .gitignore 正確設定
   - 不追蹤 venv/, node_modules/, __pycache__/
   - 不追蹤大型檔案（模型、影片）
   - 提交訊息清楚描述修改內容

3. **可以接受**：
   - 必要的註解（解釋複雜邏輯）
   - TODO 標記（如果有計畫處理）
   - 開發用的環境變數（.env.example）

#### 錯誤處理哲學
```python
# ✅ 偏好：容錯優先
try:
    result = process_video()
except Exception as e:
    logger.error(f"Error: {e}")
    return fallback_result()  # 不要讓整個系統掛掉

# ❌ 不喜歡：讓錯誤直接中斷
result = process_video()  # 沒處理錯誤，一出錯就爆炸
```

**實際案例**：FFmpeg 處理影片縮圖
- 問題：部分影片檔案不完整，FFmpeg 報錯
- 解決：加入 `-err_detect ignore_err` 和 `-fflags +genpts`
- 哲學：**盡可能提供功能，失敗時優雅降級**

### 任務執行流程（重要！）



### 問題解決模式（完整分析）

#### 階段 1：問題發現
**使用者行為**：
- 直接描述現象
- 通常會附上錯誤日誌或截圖
- 可能會用 `ultrathink` 標記複雜問題

**AI 應對**：
- 立即分析錯誤日誌
- 識別根本原因
- 準備解決方案（不要只是建議，要實際執行）

#### 階段 2：方案實施
**使用者行為**：
- 在自己的終端執行 AI 建議的指令
- 或者讓 AI 直接執行（如果是檔案操作）
- 會回報執行結果

**AI 應對**：
- 使用工具直接修改檔案
- 提供完整指令（不要省略）
- 預告預期結果

#### 階段 3：結果驗證
**使用者行為**：
- 成功：「OK」、「OK了」
- 部分成功：詳細說明哪裡不對
- 失敗：貼上新的錯誤訊息

**AI 應對**：
- 成功：進入下一步或詢問是否有其他需求
- 部分成功：分析未解決的部分，繼續修正
- 失敗：重新分析，嘗試其他方案

#### 階段 4：清理善後
**使用者行為**：
- 會主動要求：「幫我把那些臨時測試用的資訊都刪除」
- 或者 AI 應該主動清理

**AI 應對**：
- 刪除所有 console.log
- 移除測試代碼
- 整理註解
- 提醒 commit

### 當前專案：人臉辨識考勤系統

#### 系統架構
```
前端 (Vue 3 + Vite)
├── 學生註冊 (多相機拍攝)
├── 即時辨識 (多相機監控)

├── NAS API 整合 (Synology)
└── 資料管理 (JSON 檔案)
```

4. **錯誤處理**：相機斷線、NAS 連線問題、檔案損壞

|------|---------|------|

| RGB 相機縮圖失敗 | FFmpeg 錯誤容忍參數 | ✅ 已解決 |
| UI 不夠直觀 | 重新設計左右分欄布局 | ✅ 已解決 |
| 雙重 NAS 連接 | 前後端同時連接，已說明 | ✅ 已說明 |

## 🤖 AI Agent 行為準則（詳細版）

### 核心原則
1. **效率至上**：少說多做
2. **實作優先**：不要只給建議
3. **容錯設計**：考慮邊界情況
4. **主動清理**：完成後刪除測試代碼
5. **定期提醒**：功能完成後提醒 commit

### 互動檢查清單

#### ✅ 每次回應前檢查
- [ ] 是否使用繁體中文？
- [ ] 回應是否簡潔（避免冗長）？
- [ ] 是否提供完整程式碼（不用 `...` 省略）？
- [ ] 是否加上關鍵註解？
- [ ] 複雜操作是否分步驟？
- [ ] 是否實際執行而非只建議？
- [ ] 危險操作是否先確認？

#### ✅ 功能完成後檢查
- [ ] 是否刪除所有測試代碼？
- [ ] 是否刪除所有 console.log？
- [ ] 是否提醒 commit + push？
- [ ] 是否測試基本功能？

### 情境應對指南

#### 情境 1：使用者說「OK」
```yaml
行為: 立即進入下一個任務
不要: "還有其他問題嗎？" "需要我解釋嗎？"
```

#### 情境 2：使用者貼錯誤日誌
```yaml
行為:
  1. 分析錯誤原因（1-2句話）
  2. 提供解決方案
  3. 實際執行修復
不要: 只說 "你可以試試..." 而不實際修改
```

#### 情境 3：使用者說「ultrathink」
```yaml
行為:
  1. 進行全面深度分析
  2. 考慮多種可能原因
  3. 提供最佳解決方案
  4. 可以稍微詳細，但仍要結構化
不要: 簡短帶過或只給表面答案
```

#### 情境 4：使用者打斷 [Request interrupted by user]
```yaml
原因: AI 說太多了
改進: 下次更簡潔，直接進入重點
```

#### 情境 5：使用者問「沒關係嗎」
```yaml
行為:
  1. 清楚說明會發生什麼
  2. 區分 "刪除追蹤" vs "刪除檔案"
  3. 確認安全後再執行
不要: 直接說 "沒關係" 而不解釋
```

#### 情境 6：使用者說「我自己來好了」
```yaml
行為:
  1. 提供簡短提示（1-2行指令）
  2. 立即退出，不要堅持
不要: 繼續追問或堅持要幫忙
```

## 📋 常見任務與偏好

### 前端開發任務
| 任務 | 偏好做法 | 避免做法 |
|------|---------|---------|
| 修改 UI | 使用 Tailwind，現代化設計 | inline style, 老舊設計 |
| 新增組件 | Composition API, <script setup> | Options API |
| 處理狀態 | Pinia 或 reactive/ref | Vuex |
| API 呼叫 | composables (useApi.js) | 直接在組件裡寫 |
| 錯誤處理 | try-catch + 使用者友善訊息 | 讓錯誤直接爆炸 |

### 後端開發任務
| 任務 | 偏好做法 | 避免做法 |
|------|---------|---------|
| 新增 API | FastAPI + type hints | 沒有類型標註 |
| 錯誤處理 | HTTPException + 詳細訊息 | 500 Internal Error |
| 檔案操作 | pathlib.Path | 字串拼接路徑 |
| 影片處理 | subprocess + FFmpeg | 純 Python 處理 |
| 日誌記錄 | logging 模組 | print() |

### 除錯任務
| 情境 | 做法 | 不要做 |
|------|------|--------|
| 前端錯誤 | 檢查 console, network tab | 隨便猜測 |
| 後端錯誤 | 看 FastAPI 錯誤日誌 | 不看日誌就改 |
| 相機問題 | 測試 getUserMedia 權限 | 假設是程式碼問題 |
| NAS 問題 | 測試連線和 API 回應 | 假設是 NAS 壞了 |
